Boas práticas GIT para pessoas e times

[MISSING TRANSLATION] Guto Carvalho February 11, 2022 09:35:32

Em especial para Cloud Engineers.

1. Configure corretamente nome e e-mail

Para que seus commits não sejam gravados com autor “root” e-mail “root@root” configure esses valores iniciais assim que instalar seu GIT.

$ git config --global user.name "Nome Sobrenome"
$ git config --global user.email "nome.sobrenome@domain.ext"

Pode acontecer de ter que mudar de e-mail dependendo do projeto, nesse caso rode isso dentro do diretório do seu projeto para sobrescrever as configurações globais

$ cd meu_projeto/
$ git config user.name "Nome Sobrenome"
$ git config user.email "nome.sobrenome@provider.dlt"

2. Configure seu editor favorito globalmente

Eu gosto de usar o VIM

 $ git config --global core.editor vim

3. Sente com seu time e formalize o padrão de versionamento que vao seguir

Combinar antes é sai mais barato do que arrumar depois, converse com seu time e defina:

  • Convenções comuns
  • Quais serão e como usar Branchs de integrações…
  • Quais serão e como usar Branchs de features…
  • Quais serão e como usar demais branchs…
  • Padrão de nome de branchs para integração, feature, hotfix, deploy, etc…
  • Padrão esperado para commit, forma, mensagens, etc…
  • Template e forma de se fazer merge…
  • Regras gerais do que se deve ou se deve não fazer

Eu sugiro que crie um repo git e coloque tudo isso em formato MD, nada melhor do que versionar suas regras :)

4. Use e abuse de branchs

O recurso de branch é fantástico, você pode criar a partir de qualquer uma.

  • Evite gravar diretamente na main.
  • Sempre trabalhe em uma branch separada
  • Integre sua branch quando finalizar seu trabalho e finalize ela.

Exemplo

$ git branch -u main correcao_criacao_vpc
$ git checkout correcao_criacao_vpc

5. Use branchs de integração

Quando estamos trabalhando com integração contínua o uso de Branchs de integração é fundamental, precisamos integrar nosso código com o código do nosso colega e rodar os testes na pipeline.

Geralmente temos uma branch de integraçao para desenvolvimento e depois vamos promevendo esse código para outras branchs que vão nos ajudar a fechar a release e publicar, tudo depende do git-flow que seu time está seguindo.

6. Não escreva direto na MAIN

Evitamos escrever direto na MAIN pois geralmente é a última versão estável devidamente testada e valida, portanto, para colocar código lá devemos sempre passar por diversas testes em sua pipeline e revisões de seus colegas.

7. Faça commits pontuais e objetivos

Evite fazer um commit com dezenas de arquivos se esses arquivos trazem mais de uma mudança ou correção pois fica difícil rastrear, entender e revisar o que foi feito.

Prefira fazer commits pequenos e pontuais de algo que está terminado, algo que funciona e que não traga muitas alterações de uma vez só.

8. Commits atômicos quando possível

Já sabemos que commitar poucas alterações é o melhor caminho, mas dá para ser ainda menor.

Os commits atômicos são aqueles que commitam apenas uma única mudança – ainda que envolva vários arquivos – em um único commit.

É claro que nem sempre dá para fazer, mas é uma prática excelente se conseguir fazer.

A ideia do commit atômico é gravar a menor e mais importante melhoria que voce fez no código, sendo grande suficiente para adicionar algum valor, contudo, pequena o suficiente para ser gerenciada com grande flexibilidade.

Se quiser trabalhar dessa forma tenha em mente duas coisas:

  • Trabalhe em uma coisa por vez
  • Mantenha suas alterações pequenas

9. Faça commits regulares e frequentes, não espere demais

Algumas pessoas esperam demais até gravar alguma alteração, as vezes ficam ali melhorando algo que já está funcionando, buscando algum tipo de perfeição antes de gravar.

Deixa eu te dizer uma coisa, o git funciona melhor e te ajuda mais quando você grava com frequência ao invés de esperar muitoooo tempo para gravar sua mudança. Ao gravar com frequência você vai rastrear inclusive como você foi melhorando aquele código ao longo do seu processo e histórico de desenvolvimento.

E lembre-se o GIT só cuida do seu código depois que você grava, se voce fechar o editor por acidente ou se o editor travar, seu código já era, contudo, se estiver commitando com frequência você evita isso.

10. Não faça commit de algo que você não finalizou

Nunca, eu repito, nunca grave algo que não está funcionando ou que não está completo.

11. Teste seu código antes de gravar

Essa é uma das coisas mais básicas que vou falar hoje, mas tem falar e tem que fazer.

Caso não saiba, gravar código quebrado é algo bastante incômodo e poderá irritar seus pares, podendo em algum caso chegar a ser considerado uma falta de respeito e profissionalismo.

Verifique a sintaxe do seu código, rode um linter e claro, execute o código para ver se a feature ou fix estão funcionando, afinal se o código estiver ok e a alteração também, é o melhor dos mundos.

O git inclusive oferece diferentes tipos HOOKS para por exemplo executar comandos de checagem de sintaxe da linguagem e linters antes mesmo de gravar seu código, veja o link abaixo e leia sobre o hook pre-commit, é muito útil.

  • https://git-scm.com/docs/githooks

12. Escreva mensagens de commit claras e úteis para todos

12.1 Command Line

A mensagem tem que fazer sentido, ser fácil de ler e ser suficiente para entender o que foi feito.

Convenções comuns que você pode usar

Usar convenções ajudam muito, exemplo

fix: corrije bug que ocorre quando modulo vai crair a 2o VPC na AWS

outro exemplo

refact: refatora código que configura IAM para EKS na AWS

documentacao

doc: adiciona docs para uso do modulo IAM/EKS

testes

test: adiciona testes unitários no módulo

feature

feat: adiciona recurso para lidar com ALB no EC2/AWS

Explicando a estruturura

feat: added a feature to deal with ALB resources #220412
^--^  ^-----------------------------------------^
|     |
|     +-> Sumário da sua mudança
|
+-------> Tipo: docs, feat, fix, refact, ou test.

Se você estiver usando um sistema de tickets junto, adicione o número do ticket na mensagem, sistemas como GitLab e GitHub já farão a associação para você.

Evite colocar ponto final na mensagem, afinal é um título.

Vale lembrar que você deve evitar escrever mais de 50 caracteres.

De que modo escrever a mensagem?

Em alguns blogs vocês verão pessoas que defendam usar verbos no imperativo (ou presente do indicativo) e faz sentido. Em ambos os casos a ideia é que o commit responda essa pergunta:

  "Em caso de aplicar esse commit ele..."

Resposta

Altera código X
Corrige módulo Y
Refatora manifesto Z
Cria feature W para módulo X

Eu já gosto de usar particípio do passado, pois foi algo que já foi realizado e nem precisa fazer a perguntinha para ver como tem que escrever, neste caso você pode afirmar ao invés de perguntar, dessa forma:

"Nesse commit foi..."

Resposta

Alterado código X
Corrigido módulo Y
Refatorado manifesto Z
Criada feature W para módulo X

Enfim, existem longas threads sobre isso na internet, quase uma guerra santa, contudo, IMHO isso vai de depender cada um, ou da forma como cada time decidiu trabalhar.

No final o importante é você versionar seu código e conseguir entender o que fez ali só lendo a mensagem de commit :)

12.2 Usando o editor para algo mais detalhado

As vezes precisamos escrever mais, especialmente se foi uma correção complexa, se tem alguma pegadinha, ou explicar porque você fez aquilo daquela forma, no caso de Cloud Engineers nem sempre é necessário, mas se for veja como fazer.

Nesse caso não podemos fazer commit online, temos que ir para o editor.

Assunto da mensagem (no máximo 50 caracteres, sem ponto final)

Descrição em múltiplas linhas
sobre seu commit, use até 72 caracteres

Outros parágrafos devem vir após linhas em branco
Aqui nao tem mais limite de caracteres

Podemos usar bullets também, ajuda a organizar a informação

- Podemos 
- Usar
- Sempre
- Que
- Precisar
- Explicar
- Algo

Se existe um ticket coloque seu 
o número dele na mensagem

Esse commit resolve o problema do ticket #2222
E está relacionado com o ticket #3333 e #4444

13. Não reescreva o histórico da branch principal

Se você já gravou o código na branch principal, e o código alterado já está disponível para outras pessoas verem, recomenda-se não alterar esse histórico.

Fazer rebase na MAIN ou qualquer que seja o nome da sua branch upstream, ou mesmo uma branch de integração pode causar problemas para seus colegas e eles podem ter que lidar com conflitos ou fazer stash e baixar toda a branch novamente. Fora que normalmente alterações nessas acionam gatilhos em ferramentas de CICD e isso pode gerar um deploy com regressão.

Dito isso, muito cuiddo com suas branchs, em especial a upstream e as branchs de integração.

Na sua branch de feature você faz o que quiser, use o rebase sem problema algum, mas só ela ok?

14. Faça rebases regulares para manter sua feature-branch atualizada

Exemplo de atualizacao da branch main localmente e depois rebase

git checkout main
git pull
git checkout feature-xyz  # name of your hypothetical feature branch
git rebase main           # may need to fix merge conflicts in feature-xyz

Exemplo de merge do código com a main

git checkout main
git pull
git merge feature-xyz

Hoje em dia eu quase nao faço merge no braço, geralmente uso MR ou PR, depende da ferramenta que estamos usando.
